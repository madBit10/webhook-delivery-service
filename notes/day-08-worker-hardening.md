# 📘 Day 8 — Hardening + Retry Schema (getting ready to retry)

> **What I built:** the groundwork for Phase 7 (retries). Two new columns on `delivery_attempts`
> (`attempt_number`, `duration_ms`), the logic to populate them, and — most importantly — I made the
> **worker crash-proof** so one poison event can't kill delivery for everything behind it. No retry
> logic yet; this is the foundation the retries sit on.

---

## 🧱 1. Two new columns on `delivery_attempts`

```python
attempt_number = Column(Integer, nullable=False)   # which try was this? 1, 2, 3…
duration_ms    = Column(Integer, nullable=True)    # how long the outbound POST took, in ms
```

- **`attempt_number`** — once retries exist, one event has many attempts; this says *which* one each
  row is. `NOT NULL` because we always know it (today it's always `1`).
- **`duration_ms`** — observability. It's how you'd later answer "are deliveries getting slow?" or set
  an SLA. `nullable=True` so "not measured" is representable.

## ⚠️ 2. The migration gotcha — adding NOT NULL to a populated table

The autogenerated migration tried:

```python
op.add_column('delivery_attempts', sa.Column('attempt_number', sa.Integer(), nullable=False))  # ❌
```

Postgres refuses: *"what do I put in this column for the rows that already exist?"* A `NOT NULL` column
with no default can't be added to a table that already has data. Two different fixes, chosen by asking
**"is there a correct value to backfill old rows with?"**

- **`attempt_number` → YES (`1`).** Add it with a *temporary* `server_default`, backfill, then drop the
  default so the app is forced to set it going forward:
  ```python
  op.add_column('delivery_attempts',
      sa.Column('attempt_number', sa.Integer(), nullable=False, server_default='1'))
  op.alter_column('delivery_attempts', 'attempt_number', server_default=None)
  ```
- **`duration_ms` → NO honest value.** Old rows were never timed, so `NULL` = "not measured" is the
  *truthful* value. Just make it `nullable=True` — no backfill needed.

### Model ≠ migration (important)
Editing the **model** does NOT change an already-generated **migration file**. The model is what the ORM
uses at runtime + what *future* autogenerations diff against; the migration is a frozen script. If you
fix the model after generating, you must also edit the migration (or delete + regenerate). I fixed
`duration_ms` in the model but had to hand-edit the migration too.

## 🔢 3. Deriving `attempt_number` — count prior attempts

```python
def count_delivery_attempts(db: Session, event_id: int) -> int:
    return db.query(DeliveryAttempt).filter(DeliveryAttempt.event_id == event_id).count()
```

`attempt_number = count_delivery_attempts(db, event.id) + 1`. Today that's always `1`; once retries land
(Day 9) it climbs on its own — no extra counter to maintain. **Bug I caught:** first wrote
`db.query(Event).filter(DeliveryAttempt.id == event_id)` — wrong table *and* wrong column. It's the
*attempts* table, filtered on `.event_id` (the FK), not `.id` (the attempt's own PK).

## ⏱️ 4. Timing the delivery

```python
import time
...
start = time.perf_counter()
try:
    response = httpx.post(url, json=payload, timeout=5.0)
    ...
except httpx.RequestError as e:
    ...
duration_ms = int((time.perf_counter() - start) * 1000)   # seconds → ms → int
```

- **`time.perf_counter()`, not `time.time()`** — `perf_counter` is *monotonic*: immune to clock
  adjustments (NTP, DST) and can't go backwards. Exactly what you want for measuring an elapsed interval.
  `time.time()` is wall-clock and can jump.
- **Timer wraps the whole try/except** so the timeout path gets timed too (a 5s timeout records ~5000ms —
  useful signal, not noise).

## 🛡️ 5. The main event — making the worker crash-proof

Before, the per-event work sat in a bare `try/finally`. If `deliver_event` *raised* (not returned failed
— actually threw), the exception escaped the loop and **the whole worker process died.** One bad event
killed delivery for everything queued behind it.

```python
db = SessionLocal()
try:
    event = get_event(db, event_id)
    if event is None:
        print(f"Event {event_id} not found, skipping")
        continue
    deliver_event(db, event)
    print(f"Delivered event {event_id}, status={event.status}")
except Exception as e:                       # broad = last line of defense
    print(f"Error processing event {event_id}: {e}")
finally:
    db.close()                               # runs on continue AND on except — no leaked sessions
```

**Narrow low, broad high.** In the *repository* I catch the *narrow* `SQLAlchemyError` (I know exactly
what I'm handling, and re-raise). At the worker's outermost loop I catch *broad* `Exception` on purpose —
this is the last line of defense, so it must catch *everything* to guarantee the loop never dies.

## ⚠️ 7. A gap I spotted (fixed in Day 9)

When the worker crashed, events 8 & 9 were left stuck as `pending` forever — **orphaned**. Why: `BRPOP`
removes the id from the queue the instant it's popped (at-most-once). If the worker dies *between* the
pop and the status update, the id is gone from Redis but the DB row never advances, and nothing retries
it. This is **different** from a clean delivery failure (Day 9's retry logic only catches deliveries that
*ran and reported failed*, not crashes). Fix planned for Day 9: upgrade `BRPOP` → `BLMOVE` (atomically
move the id to a `processing` list; only remove after success, so a crash leaves it recoverable).
